## 给同事直接复制的 CodeBuddy 提示词

下面这些话可以直接复制给 CodeBuddy。

### 1. 首次接手仓库

```text
这是一个 pnpm workspace 的 TAI Design 组件库仓库，请你先阅读 README.md、HANDOVER.md、PROMPTS.md、WORKING-AREAS.md、package.json、pnpm-workspace.yaml、packages/docs/.env.example 以及 .codebuddy/rules 下的规则文件；然后帮我检查本机 Node 和 pnpm 是否满足要求，不满足就告诉我最少操作步骤；如果环境没问题，就安装依赖并直接启动项目（pnpm dev）。如果我缺少 packages/docs/.env.local 或 VITE_TENCENT_MAP_KEY，请明确告诉我：这不会影响大部分页面预览，只会影响地图相关示例；最后再用非技术语言告诉我：这个仓库现在应该优先在哪些目录继续维护，哪些目录只是过渡内容。
```

### 2. 日常继续做规范 / 组件

```text
请先阅读 README.md、HANDOVER.md、WORKING-AREAS.md 以及 .codebuddy/rules 下与 TAI Design 相关的规则，再理解当前仓库结构。之后我会说我的目标，你需要优先沿用现有组件、token 和规范，不要随意新增风格，不要使用不符合 6 倍数规则的间距，不要写死颜色，并始终考虑浅色与暗色模式。你先用人话告诉我今天应该在哪些目录工作、准备改哪些内容，然后再开始改。
```

### 3. 我只会说需求，不会说技术实现

```text
我只会描述设计需求，不会写代码。请你把我的自然语言需求翻译成这个仓库里的具体改动，并且在改动前先阅读相关规则文件，优先修改正式组件和正式文档，不要优先堆在临时过渡目录里。除非必须，否则不要让我自己操作命令行。
```

### 4. 新增一个正式组件时

```text
我要新增一个正式组件，请你先判断它应该放在 packages/components 的哪个位置，并同步考虑 packages/docs 里是否需要补文档或示例。请遵循 TAI Design 的 token、主题和暗色模式规范，优先复用已有模式，不要自由发挥视觉风格。完成后告诉我：组件源码改了哪里、文档改了哪里、我应该如何预览。
```

### 5. 修改已有规范页面时

```text
我要修改一个已有规范页面。请你先找出现有页面的实现位置，判断它是正式规范页还是临时过渡页；如果是正式页，就直接在正式目录修改；如果只是过渡页，请告诉我是否值得把它沉淀为正式页面。整个过程中请遵循 TAI Design 规则，不要使用硬编码间距、颜色、圆角。
```

### 6. 环境有问题时

```text
请你先自己排查并解决这个仓库的环境问题，优先帮我自动完成；只有在必须需要我登录、授权、填写密钥或者安装系统级软件时，再把需要我做的动作压缩成最少步骤告诉我。
```

### 7. 我连把项目下载到电脑都不会

```text
我刚接手这个项目，不懂 Git，也不懂“克隆仓库”是什么意思。请你把“把这个项目下载到我的电脑并打开”的步骤讲成按钮级的人话，默认使用 GitHub Desktop，不要一上来给我命令行。请你明确告诉我每一步应该点哪里、复制哪个地址、下载完后怎么打开项目，然后再继续指导我安装依赖和启动。
```
