# TAI Design 场景协作实施蓝图

## 1. 文档目的

本文档用于向研发团队说明：

- 为什么 TAI Design 需要从“组件库 + 示例页”进一步升级
- 当前做法为什么难以支撑产品与设计的持续协作
- 如果要支持“产品先生成、设计再调优、产品持续同步设计最新版本”的协作模式，推荐采用什么架构
- 这件事应该如何分阶段实施，避免一次性投入过大

这不是一份最终技术设计文档，而是一份**面向研发评审的实施蓝图**。目标是帮助团队先统一方向，再决定具体排期、边界和投入方式。

---

## 2. 背景与业务诉求

当前 TAI Design 更像一套给业务方使用的基础材料包，包含：

- 组件能力
- Token / Theme / 设计规范
- 文档站与示例页
- 若干产品工作区入口

业务上希望这套系统不只是“展示规范”，而是进一步承担下面的角色：

1. 作为业务方生成 demo 的基础底座
2. 支持产品通过自然语言或结构化方式，快速生成某个业务功能 demo
3. 支持设计在产品产出的基础上继续调优
4. 支持产品随时同步到设计已经调整过的最新版本，继续迭代
5. 支持方案逐步沉淀，而不是每次都生成一次性页面

以“随行发现”举例，希望形成的协作关系大致是：

- 产品先生成一个“随行发现”功能方案或页面原型
- 设计进入该方案，在现有结果上继续优化结构、视觉层级、状态和组件使用方式
- 产品后续再次打开时，看到的是设计已经调优后的最新版本，而不是自己最初的老版本
- 双方都围绕同一个对象持续协作，而不是互相传递静态截图或分叉代码

这意味着，系统要支撑的已经不是传统意义上的“组件库文档”，而是更接近于**可版本化、可协作的场景资产平台**。

---

## 3. 当前仓库现状判断

结合当前仓库结构，可以初步认为：

- `packages/components` 适合继续承担“基础材料包”的职责
- `packages/docs` 适合继续承担“文档站、预览站、协作入口”的职责
- 当前已有产品工作区入口，例如 `discovery`、`market`、`chat`、`connect`
- 当前产品工作区更接近“固定导入页面的预览入口”，还不是“真正可协作的场景资产系统”

也就是说，当前系统已经具备一个很好的起点，但距离真正支撑跨角色持续协作，还差一层关键能力：

> **场景定义、版本演进和发布回流机制。**

如果继续沿用“直接生成 TSX 页面，然后不同角色各自修改代码”的方式，短期虽然能跑通 demo，但中长期会非常容易出现以下问题：

- 产品生成一版页面
- 设计复制或另起一版继续修改
- 产品再改自己手上的版本
- 两边结果开始分叉
- 最终没人能明确知道哪个版本才是“当前正式方案”
- 设计调整无法自然回流给产品
- 后续所有讨论都围绕代码 diff，而不是围绕场景本身

这类问题并不只是工程实现上的麻烦，本质上是**资产模型缺失**。

---

## 4. 核心问题判断

### 4.1 需求本质不是“生成页面”，而是“协作维护同一个场景”

如果只把 demo 理解成一个 React 页面，那么生成、调整、同步、回滚都会围绕代码展开。这样会带来以下困难：

- 非研发角色难以参与
- 设计修改难以结构化记录
- 版本流转缺少统一标识
- 发布与回退成本高
- 很难做差异对比和评审

更合理的理解方式应该是：

> **产品和设计不是在协作编辑页面源码，而是在协作编辑同一个“场景对象（Scenario）”。**

### 4.2 真正要稳定的，不是 demo 代码，而是场景资产

一个可持续演进的 demo，应至少具备这些特征：

- 有稳定身份：知道自己是谁
- 有版本链路：知道每次修改从哪来
- 有状态管理：知道是草稿、评审中还是已发布
- 有发布结果：知道当前对外展示的是哪个版本
- 有对比能力：知道两个版本差在哪
- 有角色协作能力：知道产品、设计分别做了什么

所以推荐把系统设计重点，从“页面产物”切到“场景资产”。

---

## 5. 推荐目标形态

目标不是：

- 让大家同时编辑 demo 页源码

目标应该是：

- 让大家围绕统一的 `Scenario` 进行生成、调优、评审、发布和继续迭代

也就是说，整个系统建议拆成三层：

### 5.1 基础层：TAI Design 本体

职责：

- 组件
- Token
- Theme
- 设计规范
- 通用交互模式

对应现有仓库：

- `packages/components`
- `packages/docs` 中的设计系统文档页

这一层保持相对稳定，是整套系统的“材料包”。

### 5.2 场景层：Scenario DSL / Manifest

职责：

- 定义 demo 的页面结构
- 定义页面中的区块、组件、文案、交互、状态
- 作为产品与设计共同协作的核心对象

这层是当前最缺、也是最关键的一层。

### 5.3 协作层：版本、发布、对比、回滚

职责：

- 每次保存形成 revision
- 支持基于某个 revision 再编辑
- 支持 diff / compare
- 支持发布为当前正式版本
- 支持回滚到历史版本
- 支持角色维度的草稿和接力协作

## 6. 核心设计原则

### 6.1 以 Scenario 为中心，而不是以 Demo 页面代码为中心

所有角色都围绕统一的 `scenarioId` 协作，页面代码只是一种渲染结果，而不是唯一事实源。

### 6.2 先做“版本式协作”，不要一开始追求“实时协同编辑”

从业务目标看，当前真正需要的是：

- 产品先生成
- 设计在其基础上继续调整
- 产品能回到设计调整后的最新版本

这并不要求像 Figma 一样做多人实时 cursor 协同。实时协同通常意味着：

- CRDT / OT
- 实时同步服务
- 冲突解决机制
- 更复杂的权限和会话管理

对于当前阶段来说，这个复杂度明显过高。

更务实的路线是：

> **先做 revision-based collaboration（基于版本的协作），而不是 real-time collaboration（实时协作）。**

### 6.3 先做结构化场景定义，再做 AI 和可视化编辑器

如果没有统一的场景对象，AI 生成的结果和设计改动很难沉淀，也无法被版本化。

因此顺序建议是：

1. 先定义场景数据结构
2. 再做场景渲染 runtime
3. 再补版本 / 发布机制
4. 再接入可视化编辑器和 AI patch 能力

不要倒过来。

---

## 7. 推荐的整体架构

### 7.1 三层结构

```text
TAI Design 基础库
  ├─ components / tokens / theme / design rules
  └─ 提供基础材料

Scenario 场景层
  ├─ scenario manifest / DSL
  ├─ 页面结构、区块、组件、文案、交互、状态
  └─ 作为协作的单一事实源

Collaboration 协作层
  ├─ revisions
  ├─ publish
  ├─ compare
  ├─ review
  └─ 回滚与角色接力
```

### 7.2 对当前仓库的建议职责划分

#### `packages/components`
继续负责：

- 正式组件
- Token / Theme
- 通用样式与交互能力
- 长期沉淀的基础能力

#### `packages/docs`
继续负责：

- 设计规范展示
- 组件文档页
- 场景预览页
- 场景版本查看页
- 场景编辑入口（后续可扩展）

#### 建议新增的逻辑层
建议在 `packages/docs/src/` 内逐步补齐以下目录：

- `scenarios/`：场景定义、注册表、示例数据
- `scenario-runtime/`：根据场景定义渲染页面
- `scenario-editor/`：面向产品/设计的编辑入口（后续）
- `scenario-versioning/`：版本、发布、对比相关逻辑

这不一定意味着最终都放在 `docs` 包中，但从当前仓库阶段看，把它先落在 `docs` 中验证是比较务实的方式。

## 8. Scenario 数据模型建议

下面给出一份偏务实的第一版数据结构建议。目标不是一步到位，而是先形成可稳定演进的骨架。

### 8.1 Scenario 顶层字段

```json
{
  "scenarioId": "discovery-nearby-find",
  "title": "随行发现 - 附近发现",
  "productArea": "discovery",
  "route": "/discovery/nearby-find",
  "revision": "r3",
  "basedOnRevision": "r2",
  "status": "draft",
  "authorRole": "design",
  "updatedAt": "2026-03-20T10:00:00Z",
  "pages": [],
  "themeOverrides": {},
  "content": {},
  "interactions": [],
  "metadata": {}
}
```

### 8.2 推荐字段说明

- `scenarioId`
  - 场景的稳定身份标识
  - 整个协作链条都围绕它展开

- `title`
  - 场景标题

- `productArea`
  - 所属业务域，例如 `discovery`、`market`

- `route`
  - 对应预览路由

- `revision`
  - 当前版本号，例如 `r1`、`r2`、`r3`

- `basedOnRevision`
  - 当前版本是从哪个版本分支而来

- `status`
  - 建议支持：`draft` / `review` / `published` / `archived`

- `authorRole`
  - 当前版本主要由谁产出，例如：`product` / `design` / `engineering`

- `pages`
  - 页面数组

- `themeOverrides`
  - 局部主题覆盖能力

- `content`
  - 文案与业务内容映射

- `interactions`
  - 页面中的关键交互关系

- `metadata`
  - 扩展字段，如标签、优先级、来源、需求单号等

### 8.3 页面与区块结构建议

```json
{
  "id": "home",
  "name": "首页",
  "blocks": [
    {
      "id": "search-bar-1",
      "type": "searchBar",
      "props": {
        "placeholder": "搜索附近服务"
      }
    },
    {
      "id": "poi-list-1",
      "type": "poiCardList",
      "props": {
        "layout": "vertical",
        "cardStyle": "compact"
      }
    }
  ]
}
```

这类结构的意义在于：

- 产品可以通过意图生成这份结构
- 设计可以继续调整 block 排序、variant、信息层级和局部配置
- runtime 根据这份结构渲染页面
- diff 可以直接围绕结构化数据进行，而不是只对比一堆手写 JSX

## 9. 版本与发布机制建议

### 9.1 为什么版本机制是这件事的关键

如果没有版本机制，产品和设计就只能通过“拷贝页面、互相覆盖”来协作。这样很快会变成不可控的多副本系统。

### 9.2 建议的最小版本模型

每个场景至少维护以下概念：

- `scenarioId`：场景身份
- `revision`：版本号
- `basedOnRevision`：基于哪个版本修改
- `status`：当前版本状态
- `publishedRevision`：当前正式发布版本
- `latestProductDraft`：产品侧最新草稿
- `latestDesignDraft`：设计侧最新草稿

### 9.3 典型流转方式

假设有一个场景 `discovery-nearby-find`：

1. 产品生成 `r1`
2. 设计基于 `r1` 调整，产生 `r2`
3. 评审通过后，发布 `r2`
4. 系统将 `publishedRevision` 指向 `r2`
5. 产品再次进入时，默认打开 `r2`
6. 产品继续补业务逻辑，形成 `r3`
7. 后续重复以上流程

### 9.4 状态机建议

建议版本级状态不要过于复杂，第一期控制在以下几种即可：

- `draft`：草稿
- `review`：待评审
- `published`：正式发布
- `archived`：归档

如果后续需要更严格的流程，再补：

- `rejected`
- `superseded`
- `experimental`

## 10. 产品与设计的协作流建议

### 10.1 推荐工作流

```text
产品发起场景
  ↓
生成初版 Scenario（r1）
  ↓
生成预览链接
  ↓
设计基于 r1 调优，形成 r2
  ↓
对比 r1 / r2 差异
  ↓
评审通过，发布 r2
  ↓
产品基于 r2 继续迭代，形成 r3
```

### 10.2 角色分工建议

#### 产品侧更适合处理

- 业务目标定义
- 功能结构与信息架构初稿
- 页面流程
- 状态补充
- 文案方向
- 需求变更后的场景更新

#### 设计侧更适合处理

- 信息层级优化
- 区块顺序与页面节奏
- 组件选择与 variant 调整
- 视觉结构调优
- 空态、异常态、反馈态补充
- 设计规范一致性校正

#### 研发侧更适合处理

- 场景 DSL 设计
- runtime 设计与组件映射
- 版本与发布机制
- diff / review / 回滚能力
- 与现有文档站、组件库的整合

### 10.3 一个关键原则

> 产品和设计不应该互相“传 demo 文件”，而应该在同一条 revision 链上接力协作。

## 11. 对当前路由和产品工作区的改造建议

### 11.1 当前方式的问题

当前产品工作区更像是固定入口 + 固定页面导入。这样的方式在演示阶段可用，但不适合作为长期协作底座。

主要问题：

- 页面与场景身份没有解耦
- 不易版本化
- 不易做 diff / publish / rollback
- 不易让不同角色围绕统一对象接力

### 11.2 推荐改造方向

短期不要推翻现有入口，而是逐步把工作区切换为“场景驱动”。

例如从：

- `/discovery` -> 固定页面壳

逐步过渡到：

- `/discovery/:scenarioId`
- `/discovery/:scenarioId?revision=r3`

这样好处是：

- 路由天然承载场景身份
- preview link 可直接对应具体版本
- diff / compare / publish 更容易实现
- 当前产品域入口仍然可以保留

### 11.3 推荐过渡方式

第一阶段：

- 保留现有 `imports` 方案
- 新增 `scenario registry`
- 让部分页面先支持由 registry / manifest 驱动渲染

第二阶段：

- 新场景默认使用 manifest + runtime
- 老页面逐步迁移

第三阶段：

- 让“工作区入口”完全切换到场景路由模式

## 12. 建议的目录与模块划分

以下是一种适合当前仓库节奏的落地方式。

```text
packages/docs/src/
  home/
  design-system/
  imports/
  scenarios/
    registry.ts
    discovery/
      nearby-find/
        scenario.r1.json
        scenario.r2.json
        published.json
  scenario-runtime/
    renderScenario.tsx
    blockRegistry.ts
    pageRenderer.tsx
  scenario-versioning/
    revisionStore.ts
    diffScenario.ts
    publishScenario.ts
  scenario-editor/
    ScenarioWorkspace.tsx
    panels/
    hooks/
```

### 12.1 `scenarios/`

职责：

- 存放场景定义
- 存放场景注册表
- 存放已发布指针或元数据

### 12.2 `scenario-runtime/`

职责：

- 将 manifest 渲染成页面
- 将 `block.type` 映射到真实组件
- 统一处理主题、布局和页面状态

### 12.3 `scenario-versioning/`

职责：

- 版本读取
- 差异对比
- 发布逻辑
- 历史回滚

### 12.4 `scenario-editor/`

职责：

- 面向产品/设计的场景编辑界面
- 结构编辑
- 属性面板
- 版本对比和发布入口

## 13. 推荐的实施阶段

为了控制复杂度，强烈建议分阶段推进。

### 第一阶段：把 demo 升级成“可版本化场景”

#### 目标

- 定义 Scenario 数据结构
- 完成一个最小 runtime
- 支持按 `scenarioId + revision` 预览
- 支持 `publishedRevision`
- 跑通至少一个真实业务样例，例如“随行发现”

#### 第一阶段产出

- 统一的场景模型
- 一个能跑通的场景渲染链路
- 一个能支撑产品 -> 设计 -> 发布 -> 再迭代的最小闭环

#### 为什么第一阶段最重要

因为它解决的是“资产模型”和“协作基线”问题。一旦这层成立，后面的编辑器、AI、可视化能力才有稳定落点。

---

### 第二阶段：补齐编辑与评审能力

#### 目标

- 提供基础编辑器工作区
- 支持页面树 / block 树
- 支持属性面板
- 支持版本差异对比
- 支持发布记录查看

#### 这一阶段的价值

- 设计参与成本显著降低
- 评审流程更清晰
- 场景不再只能由研发手改 JSON

---

### 第三阶段：接入 AI 生成与 Patch 能力

#### 目标

- 产品通过自然语言创建场景初稿
- 根据指令生成或修改 manifest patch
- 让 AI 改的不是 JSX，而是 Scenario 结构

#### 这一阶段的好处

- AI 与协作系统天然兼容
- 所有变更都进入同一套版本系统
- 产出结果可回看、可发布、可回滚

---

### 第四阶段：视需要再评估 Figma 桥接

#### 原则

Figma 可以是设计参考来源，但不建议在第一阶段就把 Figma 作为正式单一事实源。

#### 更合适的定位

- 正式 source of truth：Scenario 系统
- Figma：视觉探索、灵感参考、设计沟通工具

这样可以避免“正式可运行版本”和“设计探索稿”长期分裂。

---

## 14. 为什么不建议一开始就做实时协同

这件事很容易让人联想到 Figma 风格协作，但从当前业务目标看，并不需要第一天就具备这个能力。

原因包括：

1. 实时协同实现成本高
2. 冲突解决复杂
3. 权限和状态同步成本高
4. 当前阶段未必能证明其投入产出比
5. 版本式协作已经足够满足主要需求

因此更建议的策略是：

- 第一阶段做“版本式协作”
- 等业务价值被验证后，再评估是否需要更细粒度的实时协作能力

## 15. 研发视角的关键模块清单

如果研发要评估工程量，可以优先拆成以下几类模块：

### 15.1 数据模型模块

- Scenario schema
- Block schema
- Page schema
- Revision metadata
- Publish metadata

### 15.2 Runtime 模块

- block registry
- scenario renderer
- page renderer
- theme adapter
- state renderer

### 15.3 协作模块

- revision store
- compare / diff
- publish pointer
- rollback
- draft management

### 15.4 工作区模块

- scenario explorer
- preview panel
- properties panel
- revision history
- publish action

### 15.5 后续智能能力模块

- natural language to manifest
- manifest patch generation
- structured change explanation

## 16. 工程量判断与务实建议

从直觉上看，这件事容易被理解成一个很大的平台项目。但如果控制边界，其实可以拆成一个相对可落地的演进路径。

### 不建议的做法

- 一开始就做全量平台
- 一开始就做多人实时编辑
- 一开始就做完整低代码系统
- 一开始就追求 Figma 双向同步

### 建议的做法

- 先拿一个真实业务场景跑通
- 先验证场景 DSL 是否足够表达目标页面
- 先把“版本、发布、回流”机制做成最小闭环
- 等协作链条真实跑通后，再决定是否继续建设编辑器和 AI 能力

也就是说，这件事虽然方向上像平台，但落地时应当按**“单场景试点 -> 形成通用骨架 -> 再逐步平台化”**的方式推进。

## 17. 风险与关键取舍

### 17.1 风险一：Scenario DSL 设计过重

如果 DSL 一开始就追求覆盖所有页面类型，很容易过度设计。

#### 建议

- 从 1~2 个高频业务样例出发设计
- 先覆盖核心 block 类型
- 保留扩展点，不追求一步到位

### 17.2 风险二：Runtime 表达力不足

如果 runtime 过于抽象，可能会影响页面还原能力。

#### 建议

- 允许部分 block 先绑定具体业务组件
- 在可复用和表达力之间做阶段性平衡

### 17.3 风险三：设计和产品的协作习惯没切换过来

如果工作流仍然围绕截图、Figma 页面或独立代码文件进行，Scenario 体系会失去价值。

#### 建议

- 强制通过 `scenarioId` 和 `revision` 协作
- 让预览链接、评审链接、发布记录都围绕同一对象展开

### 17.4 风险四：过早进入编辑器建设

编辑器容易消耗大量前端资源。

#### 建议

- 第一阶段先允许研发维护 JSON / TS manifest
- 确认模型稳定后，再建设面向产品和设计的可视化编辑能力

---

## 18. 推荐的首期试点方案

建议不要同时覆盖所有业务域，而是选择一个代表性场景试点。

### 推荐试点对象

- `discovery` 业务域中的一个核心页面或功能流

### 首期试点目标

- 选定一个真实场景
- 定义其 Scenario manifest
- 跑通 runtime 渲染
- 跑通 revision 和 publish
- 让产品与设计完成至少一轮接力协作

### 首期试点成功标准

- 产品可以生成或发起一个场景初稿
- 设计可以基于它继续调整
- 发布后产品再次进入能看到设计最新版本
- 整个过程不依赖多份分叉页面源码

如果这个闭环能跑通，就说明方向基本成立。

## 19. 研发评审时建议重点讨论的问题

为了提高讨论效率，建议研发团队优先围绕以下问题给出判断：

1. 是否认同“Scenario 作为单一事实源”的方向
2. 第一阶段的 DSL 设计边界应该如何控制
3. runtime 放在 `docs` 包中验证是否合适
4. revision / publish 元数据先存仓库，还是直接上服务端存储
5. 首个试点场景选哪个最合适
6. 第一阶段哪些能力必须做，哪些可以延后
7. 现有 `imports` 内容如何平滑过渡

## 20. 结论

如果 TAI Design 的目标只是“展示组件和规范”，那么现有结构已经基本够用。

但如果目标是：

- 给业务方作为基础材料包
- 支持产品快速生成 demo
- 支持设计持续调优
- 支持版本回流和继续迭代
- 支持长期沉淀为可复用场景资产

那么 TAI Design 应该进一步升级为：

> **组件库 + 场景资产系统 + 版本化协作机制**

最核心的设计判断是：

> **不要把 demo 当成一次性页面代码，而要把它当成可版本化、可发布、可协作的 Scenario 资产。**

这件事方向上确实不小，但并不意味着必须一次性做成重平台。更合理的路线是：

- 先定义 Scenario
- 再做 runtime
- 再补版本 / 发布
- 然后再做编辑器和 AI

先用一个真实业务场景跑通最小闭环，再决定后续平台化投入。

---

## 21. 给研发的简版一句话摘要

建议把 TAI Design 从“组件库 + 示例页”升级成“组件库 + 场景资产系统 + 版本化协作流”，并以 `Scenario` 作为产品、设计、研发共同协作的核心对象，而不是继续围绕分散的 demo 页面源码协作。
